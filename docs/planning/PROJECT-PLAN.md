# Project Plan: CYO Adventure (Ariadne)

> **Status**: Active | **Version**: 1.2 | **Updated**: 2026-07-01
> **Codename**: Ariadne (the thread that guides a reader through the maze of choices)
> **Primary branch**: `main`

## 1. Executive Summary

CYO Adventure is a family app that plays branching "choose your path" gamebook stories offline
on a tablet or phone, paired with a content pipeline that turns a short concept plus a drafting
guide into a finished, safety-reviewed story. It is built for one family's four children
(roughly ages 9 to 15), and no story reaches a child until a parent approves it.

The system has two halves that meet at a single file format: a **reader** plays stories and a
**content pipeline** produces them. Both treat a story as one artifact, a versioned JSON graph
of passages and choices with optional state for older kids. The pipeline runs staged LLM
generation (structure, then prose, then self-repair) with a deterministic validator between
stages and a mandatory parent approval step before any story reaches a child's library.

**Key innovation**: a hard validation gate (graph integrity plus state-space analysis) plus
mandatory guardian approval means correctness and safety are structural guarantees, not
conventions.

**Decided release cut**: Generation ships in the **first usable release**. Phases 0 through 3
plus the minimal Phase 4a library-and-profiles slice constitute the first usable release
(roughly 11 to 16 weeks for a 1 to 2 developer team). Phase 4b (node editor, ending tracker,
read-aloud) and Phase 5 (hardening and restore drill) follow the first release.

Because generation ships in the first release, two items are elevated to **Phase-0 hard
blockers**: the LLM provider data-handling decision and the privacy controls (no real child PII
in prompts; admin-only short-lived raw outputs).

**Timeline**:

| Band | Scope | Estimate |
|------|-------|----------|
| First usable release | Phases 0-3 + Phase 4a | 11-16 weeks |
| Full v1 | All phases including 4b and 5 | 16-25 weeks |

## 2. Scope

### In scope (MVP and first release)

- **Reader (PWA)**: offline play, save and resume, multi-device sync, multiple endings.
  ([ADR-002](./adr/adr-002-client-pwa.md))
- **JSON Storybook format**: Tier 1 (simple branching) and Tier 2 (small state-tracking).
  ([ADR-001](./adr/adr-001-story-format-json-storybook.md))
- **Staged LLM generation**: concept brief to finished story through structure, prose, and repair
  passes behind a provider-agnostic interface. ([ADR-003](./adr/adr-003-frontier-llm-generation.md))
- **Validation gate**: deterministic Layer-1 graph checks plus Layer-2 state-space analysis.
- **Content-moderation gate**: provider moderation plus an independent LLM-reviewer pass, scored
  per age band.
- **Parent approval workflow**: publish state machine enforcing guardian approval before any story
  is visible to a child. ([ADR-005](./adr/adr-005-mandatory-human-approval.md))
- **Per-child profiles**: age-band and reading-level limits enforced by the library API.
- **Homelab deployment**: behind Pangolin (zero-trust) and Authentik (OIDC), with cloud-portable
  containers. ([ADR-004](./adr/adr-004-homelab-first-deployment.md))

### Out of scope (v1)

- **Dice combat and skill/luck checks (Tier 3)**: randomness makes "every path reaches an
  ending" unprovable by the deterministic gate.
- **Social, chat, or user-to-user features**: a children's app has no reason to carry one.
- **Monetization, accounts for outside users, or telemetry on minors.**
- **Native iOS/Android apps**: the JSON format keeps this open for the future.
- **Rich visual graph canvas editor**: deferred post-v1; a lightweight node editor lands in
  Phase 4b.
- **Sharing beyond this family**: would require a revisited privacy posture (COPPA, age
  assurance, verifiable parental consent).
- **Per-passage illustrations or AI-generated images.**

## 3. Architecture Overview

The system is a **modular monolith backend** (FastAPI) with an asynchronous RQ worker for
generation, and a single role-aware PWA client. Story blobs live in object storage (MinIO,
S3-compatible); metadata and reading state live in Postgres; the Redis queue drives background
generation.

Key architectural decisions, each recorded in an ADR:

| ADR | Decision | Rationale summary |
|-----|----------|-------------------|
| [ADR-001](./adr/adr-001-story-format-json-storybook.md) | Custom versioned JSON Storybook schema | Most reliable LLM target; static safety properties are checkable; client-agnostic |
| [ADR-002](./adr/adr-002-client-pwa.md) | PWA (React 19, TypeScript, Vite, service worker, IndexedDB) | One codebase, offline-capable, no app-store friction, client-agnostic format keeps native open |
| [ADR-003](./adr/adr-003-frontier-llm-generation.md) | Anthropic Claude primary behind a `GenerationProvider` interface; Ollama and OpenRouter as fallback | Frontier models hold branching structure best; provider interface keeps switching cheap |
| [ADR-004](./adr/adr-004-homelab-first-deployment.md) | Homelab-first (Pangolin, Authentik, Postgres, Redis, MinIO); cloud-portable containers | Minors' data stays on controlled hardware; existing infra reused; Azure Container Apps is a drop-in |
| [ADR-005](./adr/adr-005-mandatory-human-approval.md) | Publish state machine with mandatory guardian approval; no auto-publish path | Automated moderation helps but cannot be the sole safeguard for machine-generated content read by children |
| [ADR-006](./adr/adr-006-conditions-inhouse-evaluator.md) | JSONLogic shape, in-house whitelisted evaluator (Python + TypeScript, 10 operators) | Tiny, exhaustively tested, no dependency, no supply-chain risk; validator and player guaranteed identical semantics |
| [ADR-008](./adr/adr-008-first-release-trust-boundary.md) | First-release trust boundary: OIDC verification at the API (C4a-0), admin-inherits-guardian role model, offline-tolerant PWA sessions | On a home LAN the ingress can be bypassed; the API signature check is the only boundary that holds against the in-family threat model |

### Publish state machine

As merged (`publishing/state_machine.py` is the single source of truth): five resting
states; generation and auto-check happen while the row rests in `draft`, and `approved` is
collapsed into the `approve` action rather than being a distinct state.

```text
draft --submit------> in_review --approve---> published --archive--> archived
  |                    |    ^
  +--auto_reject--+    |    |
                  v    v    |
                needs_revision --submit--(re-review after repair / regenerate)
```

A story is visible to a child only in `published` (with a recorded approver as a library
backstop). The `approve` action requires the **global admin** role (ADR-005 as amended
2026-06-30; the approving parent is provisioned as admin per ADR-008) and stamps
`approved_by` and `published_at` in the same transition. Moderation failures route to
`needs_revision` via `auto_reject`; clean or repaired stories `submit` to review.

the merged Phase 3 shape.

## 5. Phased Development

### Phase overview and release cut

```text
Phase 0: Foundations        (1-2 wks)   chore/phase-0-foundations
Phase 1: Schema + Reader    (3-5 wks)   feat/phase-1-schema-reader
Phase 2: Gen + Gate         (4-6 wks)   feat/phase-2-generation-gate
Phase 3: Safety + Review    (3-4 wks)   feat/phase-3-safety-review
Phase 4a: Library           (part of 3-5 wks)  feat/phase-4a-library-profiles  <- FIRST RELEASE
Phase 4b: Editor + UX       (post-release)     feat/phase-4b-editor-ux
Phase 5: Hardening          (2-3 wks)   chore/phase-5-hardening
```

**Critical path**: Schema (Phase 0) -> Player + Reader + Layer-1 validator (Phase 1) -> Layer-2
state-space validator (Phase 2) -> Generation (Phase 2) -> Safety + review (Phase 3) ->
Library/profiles (Phase 4a). The schema is the keystone: settle and version it before writing
any app code. The honest long pole is Phase 2, where generation reliability and the state-space
validator absorb most of the iteration.

### Phase 2: Validation Gate and Authoring Pipeline (4-6 weeks)

**Branch**: `feat/phase-2-generation-gate` (PRs #5, #6) plus Phase 2b (PRs #7, #8)

**Objective**: Generate stories that hold together, with the gate as the arbiter. This is the

**Deliverables** (traced to [Roadmap Phase 2](./roadmap.md)):

  - Stateful dead-end detection (the "silver door" case).
  - Stateful termination and loop-escape verification.
  - Conditional-usefulness check (dead branches flagged).
  - Configuration cap: fail with "state space too large" if reachable set exceeds 100,000.
- **Generation orchestrator** with staged passes:
  - Stage A (Structure): skeleton JSON (node IDs, beats, choice edges, variable declarations,

  - Stage B (Prose): full Storybook JSON at target reading level, second person; validate again.
  - Stage C (Repair): named failing node IDs and specific violations; cap at 3 attempts;
    no-progress abort (report or output hash unchanged); on exhaustion route to full regeneration

- **Provider interface** (`GenerationProvider`): Claude primary; Ollama and OpenRouter as fallback
  ([ADR-003](./adr/adr-003-frontier-llm-generation.md)).
- **Concept intake**: concept brief fields as defined in [Tech Spec](./tech-spec.md#authoring-pipeline-staged-generation);
  no real child PII in prompts.
- **RQ worker queue** for async generation.
- **Known-bad corpus** and **Tier-2 state corpus** with tests covering: silver-door dead end,
  state-reachable trap cycle, unsatisfiable conditional, bound-overflow transition,
  configuration-cap blowup.

- From a concept brief, the pipeline produces a story that passes the full gate with zero

## 6. Git Branch Strategy

Branch names follow `{type}/phase-{N}{suffix}-{description}`. The primary branch is `main`.

| Phase | Branch | Type rationale |
|-------|--------|----------------|
| Phase 0 | `chore/phase-0-foundations` | Infrastructure and decision lock; no user-facing feature |
| Phase 1 | `feat/phase-1-schema-reader` | New user-facing capability (reader and schema) |
| Phase 2 | `feat/phase-2-generation-gate` | New user-facing capability (generation pipeline) |
| Phase 3 | `feat/phase-3-safety-review` | New user-facing capability (approval workflow) |
| Phase 4a | `feat/phase-4a-library-profiles` | New user-facing capability (library and profiles) |
| Phase 4b | `feat/phase-4b-editor-ux` | New user-facing capability (editor, TTS, tracker) |
| Phase 5 | `chore/phase-5-hardening` | Infrastructure hardening; no new features |

### Branch workflow

```bash
# Start a phase
git checkout -b feat/phase-1-schema-reader

# Or using the milestone skill
/git/milestone start feat/phase-1-schema-reader

# Complete a phase: create PR, merge via signed conventional commit, start next phase
```

Each phase branch merges to `main` via a PR after all acceptance criteria and quality gates
pass. No direct pushes to `main`.

## 7. Quality Gates (Definition of Done)

These gates apply to every phase. Phase-specific gates are listed in each phase section above.

| Gate | Standard | Enforcement |
|------|----------|-------------|
| Line coverage | >= 80% | `pytest --cov-fail-under=80` in CI |
| Branch coverage | >= 70% | `pytest --cov-branch` in CI |
| Critical paths and patch coverage | >= 90% | Condition evaluator, player state machine, validator, state machine, auth |
| Linting | Ruff: zero errors | `uv run ruff check .` |
| Formatting | Ruff format | `uv run ruff format --check .` |
| Type checking | BasedPyright strict: zero errors | `uv run basedpyright src/` |
| Frontend linting | ESLint + Prettier clean | CI step |
| Security: SAST | No high or critical | Bandit, Semgrep managed rules, CodeQL |
| Security: dependencies | No high or critical | `uv run pip-audit`; OSV-Scanner |
| Security: containers | No high or critical | Trivy |
| Secret scanning | Clean | detect-secrets |
| SBOM | Generated | CycloneDX; images signed with Cosign |
| Pre-commit | Green | `pre-commit run --all-files` before every commit |
| Commits | Signed + conventional | `git commit -S`; conventional commit message format |

A feature is complete when code is reviewed and approved, tests are written and passing at the
thresholds above, documentation is updated, no linting or type errors remain, security scans
show no high or critical findings, and the commit is signed and merged to `main` via a PR.

Source: [Roadmap: Definition of Done](./roadmap.md#definition-of-done) and project CLAUDE.md.

## 9. Success Metrics

Source: [Project Vision](./project-vision.md) section "Success Metrics".

| Metric | Target | Verification |
|--------|--------|-------------|
| Reader correctness | 100% of published stories play to at least one ending with no dead ends or broken choices | Validator rejects all invalid stories before publish |
| Generation yield | >= 60% of generated stories pass the full validation gate with zero structural edits (prose tweaks allowed) | Measured over a 20-story sample in Phase 2 |
| Defect capture | Validator rejects 100% of the curated known-bad corpus (dangling targets, orphan nodes, unreachable endings, unsatisfiable conditions, reading-level misses, unsafe content) | Known-bad corpus tests in Phase 1 (Layer 1) and Phase 2 (Layer 2) |
| Safety | 0 stories reach a child profile without a recorded parent approval | Enforced by the publish state machine; transition tests in Phase 3 |
| Offline reliability | A downloaded story plays start to finish with the network disabled | Playwright offline E2E test in Phase 1 |
