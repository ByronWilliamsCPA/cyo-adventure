Conventions for the PlantUML architecture diagrams under `docs/architecture/diagrams/`.
Following them keeps the diagram set visually consistent, machine-renderable, and
traceable back to source. The catalog of diagrams lives in [INDEX.md](INDEX.md); the
shared styling primitives live in [style.puml](style.puml).

## File and naming rules

- **One diagram per file**, named `<kind>-<subject>.puml` (for example
  `component-generation.puml`, `seq-device-grant.puml`, `journey-kid.puml`).
- **The `@startuml` name MUST match the filename stem.** The SVG generator derives
  each output path from the source stem, and PlantUML names its output after the
  `@startuml` name. A mismatch makes the renderer write `foo.svg` while committing a
  diagram whose `@startuml` says `bar`, so the freshness check and the file on disk
  disagree. Example: `seq-offline.puml` must start with `@startuml seq-offline`.
- **Every diagram has a paired SVG** (`<stem>.svg`) committed alongside it. The two
  are a unit; never commit a `.puml` change without its regenerated `.svg`.
- **Auto-generated skeletons are off-limits.** Everything under `diagrams/skeletons/`
  is generated by `scripts/render_skeleton_diagrams.py` from skeleton JSON. Do not
  hand-edit those; change the skeleton source and regenerate.

## Shared styling

New diagrams should pull shared skinparams and colour constants from `style.puml`
instead of copy-pasting them:

```text
@startuml component-example
!include style.puml
rectangle "Backend" as be $backend_green
...
@enduml
```

The existing 20 top-level diagrams still inline these values by hand. Retrofitting
them to `!include` would re-render every SVG for no visual change, so they are left
as-is; when you touch one, keep its inline values in sync with `style.puml`.

### Domain palette

Fill colours carry meaning. A reader learns this legend once and reads it everywhere.

| Constant | Hex | Meaning |
| --- | --- | --- |
| `$kid_blue` | `#E3F2FD` | Child / PWA surface (container fill) |
| `$kid_blue_sub` | `#DDEEFF` | Child / PWA sub-component fill |
| `$backend_green` | `#E8F5E9` | Backend / guardian surface (container fill) |
| `$backend_sub` | `#CCEECC` | Backend sub-component fill |
| `$backend_teal` | `#B2DFDB` | Backend runtime process (e.g. the uvicorn app) |
| `$admin_purple` | `#EDE7F6` | Admin / global safety operator |
| `$build_grey` | `#E0E0E0` | Build output or external / managed system |
| `$store_amber` | `#FFF3E0` | Data store (Postgres / Redis / object storage) |

### Canvas defaults

Font `Roboto`, background `#FAFAFA`, `roundCorner 8`, shadowing off. These are set once
in `style.puml`.

### Status stereotypes

Mark anything **not yet shipped** so "planned" reads as a deliberate decision, not an
accidental omission, and so a later audit can find it:

- `<<planned>>` (rose): designed but not built.
- `<<unverified>>` (amber): present in a diagram but not yet confirmed against code.

Verify a stereotyped element against the codebase before removing the mark. Negative
and provisional claims rot exactly when the project succeeds at building the thing they
deny, so they need re-checking on every audit.

## Traceability notes

Every hand-authored diagram carries a source-file map so a reader can jump from a box to
the code behind it, and so an audit can detect drift. Use a comment block just after
`@startuml`:

```text
@startuml component-example
' Source files:
'   src/cyo_adventure/example/service.py   (what this box represents)
'   src/cyo_adventure/example/models.py    (the data it owns)
```

Keep these paths current: a renamed or deleted source file listed here is a stale
reference the diagram audit will flag.

## PlantUML gotchas

- **No trailing `'` comment on a `!$` line.** PlantUML's preprocessor swallows an
  end-of-line comment into the value and fails the render. Put the comment on its own
  line above the assignment. (This is why `style.puml` documents each colour in a block,
  not inline.)
- **`'` line comments only work inside a diagram.** Text before the first `@startuml`
  is not comment-parsed, so a literal `@startuml` token in a header comment starts a
  spurious diagram. Include-only files (like `style.puml`) must therefore avoid the
  literal token entirely; the generator skips any `.puml` with no real diagram-start
  line.

## Regenerating SVGs

Use the generator; do not invoke PlantUML by hand (the tool resolves and SHA-256-verifies
the pinned jar for you):

```bash
# Re-render only diagrams whose SVG is missing or older than the source
python tools/generate_diagram_svgs.py

# Force a full re-render
python tools/generate_diagram_svgs.py --all

# CI / pre-push freshness gate: exit non-zero if any SVG is stale, render nothing
python tools/generate_diagram_svgs.py --check
```

The tool renders each diagram individually and derives its output path from the source
stem, so it is immune to the "newest file in the directory, then rename" corruption that
can silently swap two diagrams' SVGs. After a `--all` run it also fails loudly if any two
sibling SVGs come out byte-identical, which would signal exactly that class of bug.

The pinned renderer is **PlantUML 1.2024.7** (SHA-256 verified). The jar is resolved from
`$PLANTUML_JAR`, then a version-pinned cache at `~/.cache/cyo-adventure/`, then a one-time
download; a missing or unverified jar degrades to a clear error rather than an unverified
execution. Skeleton diagrams share the same pinned jar via
`scripts/render_skeleton_diagrams.py`.
