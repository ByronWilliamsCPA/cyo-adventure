    Project Plan: CYO Adventure (Ariadne)¶

    Status: Active | Version: 1.0 | Updated: 2026-06-20 Codename: Ariadne (the thread that guides a reader through the maze of choices) Primary branch: main

    1. Executive Summary¶

    CYO Adventure is a family app that plays branching "choose your path" gamebook stories offline on a tablet or phone, paired with a content pipeline that turns a short concept plus a drafting guide into a finished, safety-reviewed story. It is built for one family's four children (roughly ages 9 to 15), and no story reaches a child until a parent approves it.

    The system has two halves that meet at a single file format: a reader plays stories and a content pipeline produces them. Both treat a story as one artifact, a versioned JSON graph of passages and choices with optional state for older kids. The pipeline runs staged LLM generation (structure, then prose, then self-repair) with a deterministic validator between stages and a mandatory parent approval step before any story reaches a child's library.

    Key innovation: a hard validation gate (graph integrity plus state-space analysis) plus mandatory guardian approval means correctness and safety are structural guarantees, not conventions.

    Decided release cut: Generation ships in the first usable release. Phases 0 through 3 plus the minimal Phase 4a library-and-profiles slice constitute the first usable release (roughly 11 to 16 weeks for a 1 to 2 developer team). Phase 4b (node editor, ending tracker, read-aloud) and Phase 5 (hardening and restore drill) follow the first release.

    Because generation ships in the first release, two items are elevated to Phase-0 hard blockers: the LLM provider data-handling decision and the privacy controls (no real child PII in prompts; admin-only short-lived raw outputs).

    Timeline:

    Band Scope Estimate
    First usable release Phases 0-3 + Phase 4a 11-16 weeks
    Full v1 All phases including 4b and 5 16-25 weeks

    Source: Project Vision sections 1-3.

    2. Scope¶

    In scope (MVP and first release)¶

    • Reader (PWA): offline play, save and resume, multi-device sync, multiple endings. (ADR-002)
    • JSON Storybook format: Tier 1 (simple branching) and Tier 2 (small state-tracking). (ADR-001)
    • Staged LLM generation: concept brief to finished story through structure, prose, and repair passes behind a provider-agnostic interface. (ADR-003)
    • Validation gate: deterministic Layer-1 graph checks plus Layer-2 state-space analysis.
    • Content-moderation gate: provider moderation plus an independent LLM-reviewer pass, scored per age band.
    • Parent approval workflow: publish state machine enforcing guardian approval before any story is visible to a child. (ADR-005)
    • Per-child profiles: age-band and reading-level limits enforced by the library API.
    • Homelab deployment: behind Pangolin (zero-trust) and Authentik (OIDC), with cloud-portable containers. (ADR-004)

    Out of scope (v1)¶

    • Dice combat and skill/luck checks (Tier 3): randomness makes "every path reaches an ending" unprovable by the deterministic gate.
    • Social, chat, or user-to-user features: a children's app has no reason to carry one.
    • Monetization, accounts for outside users, or telemetry on minors.
    • Native iOS/Android apps: the JSON format keeps this open for the future.
    • Rich visual graph canvas editor: deferred post-v1; a lightweight node editor lands in Phase 4b.
    • Sharing beyond this family: would require a revisited privacy posture (COPPA, age assurance, verifiable parental consent).
    • Per-passage illustrations or AI-generated images.

    Source: Project Vision section "Scope Definition".

    3. Architecture Overview¶

    The system is a modular monolith backend (FastAPI) with an asynchronous RQ worker for generation, and a single role-aware PWA client. Story blobs live in object storage (MinIO, S3-compatible); metadata and reading state live in Postgres; the Redis queue drives background generation.

    Key architectural decisions, each recorded in an ADR:

    ADR Decision Rationale summary
    ADR-001 Custom versioned JSON Storybook schema Most reliable LLM target; static safety properties are checkable; client-agnostic
    ADR-002 PWA (React 19, TypeScript, Vite, service worker, IndexedDB) One codebase, offline-capable, no app-store friction, client-agnostic format keeps native open
    ADR-003 Anthropic Claude primary behind a GenerationProvider interface; Ollama and OpenRouter as fallback Frontier models hold branching structure best; provider interface keeps switching cheap
    ADR-004 Homelab-first (Pangolin, Authentik, Postgres, Redis, MinIO); cloud-portable containers Minors' data stays on controlled hardware; existing infra reused; Azure Container Apps is a drop-in
    ADR-005 Publish state machine with mandatory guardian approval; no auto-publish path Automated moderation helps but cannot be the sole safeguard for machine-generated content read by children
    ADR-006 JSONLogic shape, in-house whitelisted evaluator (Python + TypeScript, 8 operators) Tiny, exhaustively tested, no dependency, no supply-chain risk; validator and player guaranteed identical semantics

    Publish state machine¶

    draft -> generating -> auto_check -+-> needs_revision -> (repair / regenerate) -+
                                   |                                             |
                                   +-> in_review -> approved -> published -> archived
                                          ^
                                          +-- (re-review on edit)

    A story is visible to a child only in published. The transition in_review -> approved requires a guardian. Failures from the validator or moderation route to needs_revision.

    Source: Tech Spec sections "Architecture" and "Data Model".

    5. Phased Development¶

    Phase overview and release cut¶

Phase 2: Gen + Gate         (4-6 wks)   feat/phase-2-generation-gate

The honest long pole is Phase 2, where generation reliability and the state-space validator absorb most of the iteration.

    Source: Roadmap sections "Critical Path" and "Timeline Overview".

    Phase 0: Foundations (1-2 weeks)¶

    Branch: chore/phase-0-foundations Milestone: M0 - Phase 0 exit gate (decisions locked, CI green) Status: Planned

    Objective: Lock the decisions and artifacts that are expensive to change once code exists. No app code until this gate passes.

    Phase-0 hard blockers (elevated because generation ships in the first release):

    1. Provider data-handling decision: confirm with the provider whether the standard API retention path or a zero-data-retention path applies. Document the outcome; do not assume from scoping documents.
    2. Privacy controls: establish that no real child PII (name, birthdate, sensitive trait) can enter a prompt; that raw LLM outputs and prompt text are admin-only and short-lived; and that moderation reports persist with the story version for audit.

    Phase 1: Schema, Runtime, and Reader MVP (3-5 weeks)¶

    Branch: feat/phase-1-schema-reader-v2 Milestone: M1 - Reader plays hand-authored stories offline Status: Implemented (PR pending review). Condition evaluator (Python + TS, shared conformance), Layer-1 validator, deterministic player (Python + TS, cross-impl trace conformance), reader API (reading-state with 409 reconciliation, IDOR-tested), PWA reader (XState, IndexedDB, offline sync, conflict UX), Playwright E2E, and two hand-authored stories are complete and pass locally; PR review and CI are in progress.

    Objective: Prove the format and the player with human-written stories before any LLM is involved. This phase has no external network egress. See ADR-001 and ADR-002.

    Deliverables (traced to Roadmap Phase 1):

    • Deterministic player library: node traversal, state effects per Runtime Semantics v1, in-house condition evaluator (Python + TypeScript), cross-implementation conformance fixtures passing (ADR-006).
    • PWA reader: state-gated choices (hidden when condition is false, not shown-and-disabled), offline caching (service worker + IndexedDB), save/resume, multi-device sync (revision-based, 409 reconciliation).
    • Offline-conflict UX: the 409 "continue from this device" vs "use newer progress" dialog (copy plus wireframe) and the iOS post-eviction "download needed" state, designed before the Playwright reconciliation test is written so the test asserts the real UX.
    • Layer-1 graph validator with the valid/invalid fixture corpus from Phase 0 (schema, reference integrity, reachability, termination, no trap loops, condition/effect consistency, length budget).
    • Two hand-authored stories: one Tier 1 (8-11 band) and one Tier 2 (older band).

    User stories (from Roadmap Phase 1):

    • US-101: A child reads a downloaded story to multiple endings with the network disabled.
    • A previously downloaded story plays start to ending with the network disabled.
    • Reaching an ending records a completion that syncs on reconnect.
    • US-102: A middle-band reader sees state-gated choices appear only when valid.
    • A choice with a false condition is hidden, not shown-and-disabled.
    • The player and validator agree on the condition's value (conformance fixtures pass).

    Acceptance criteria (from Roadmap Phase 1):

    • A child reads a downloaded story to multiple endings with the network disabled.
    • State-gated choices appear and resolve correctly under different variable states.
    • Progress survives reopening the app; a two-device conflict resolves without silent loss.
    • The same fixtures play identically in the test harness and the browser.

    Quality gates:

    • 80% line / 70% branch coverage overall; 90% on the condition evaluator and player state machine (critical paths)
    • Cross-implementation conformance corpus passes through both evaluators with identical results
    • Runtime semantics fixtures: transition order, once on re-entry, bound rejection, hidden-vs-shown choices, version pinning all pass identically in test harness and browser
    • Playwright E2E: offline playthrough, save/resume, 409 reconciliation
    • No high or critical Bandit, pip-audit, or Semgrep findings
    • Ruff clean; BasedPyright strict clean; ESLint + Prettier clean
    • Pre-commit green; all commits signed with conventional messages

    Dependencies: Requires Phase 0 schema and scaffold. Blocks Phase 2.

    Phase 2: Validation Gate and Authoring Pipeline (4-6 weeks)¶

    Branch: feat/phase-2-generation-gate Milestone: M2 - Concept-to-story pipeline passes the full gate Status: Planned

    Objective: Generate stories that hold together, with the gate as the arbiter. This is the long pole of the project: generation reliability and the state-space validator absorb most of the iteration. First external LLM call, so the privacy controls and provider data-handling decision from Phase 0 are preconditions. See ADR-003.

    Deliverables (traced to Roadmap Phase 2):

    • Layer-2 state-space validator (Tier-2 stories only):
    • Configuration walk: start at (start_node, initial var_state); compute visible choices; explore the closure.
    • Stateful dead-end detection (the "silver door" case).
    • Stateful termination and loop-escape verification.
    • Conditional-usefulness check (dead branches flagged).
    • Configuration cap: fail with "state space too large" if reachable set exceeds 100,000.
    • Generation orchestrator with staged passes:
    • Stage A (Structure): skeleton JSON (node IDs, beats, choice edges, variable declarations, ending placement); validate graph before spending tokens on prose.
    • Stage B (Prose): full Storybook JSON at target reading level, second person; validate again.
    • Stage C (Repair): named failing node IDs and specific violations; cap at 3 attempts; no-progress abort (report or output hash unchanged); on exhaustion route to full regeneration or human review. Never auto-publish.
    • Provider interface (GenerationProvider): Claude primary; Ollama and OpenRouter as fallback (ADR-003).
    • Concept intake: concept brief fields as defined in Tech Spec; no real child PII in prompts.
    • RQ worker queue for async generation.
    • Known-bad corpus and Tier-2 state corpus with tests covering: silver-door dead end, state-reachable trap cycle, unsatisfiable conditional, bound-overflow transition, configuration-cap blowup.

    Acceptance criteria (from Roadmap Phase 2):

    • From a concept brief, the pipeline produces a story that passes the full gate with zero structural edits at least 60% of the time over a 20-story sample.
    • The validator rejects 100% of the known-bad and Tier-2 corpora with correct rule and node attribution.
    • No prompt sent to the provider contains a real child name, birthdate, or sensitive trait.

    Quality gates:

    • 80% line / 70% branch coverage overall; 90% on the Layer-2 validator and generation orchestrator (critical paths)
    • Layer-2 corpus (silver door, trap cycle, unsatisfiable condition, bound overflow, config-cap blowup) all rejected with correct rule attribution
    • Integration tests: pipeline with mocked provider returning canned and malformed outputs, proving repair loop and no-progress abort
    • No high or critical security findings
    • Ruff clean; BasedPyright strict clean
    • Pre-commit green; all commits signed with conventional messages

    Dependencies: Requires Phase 1 format, player, and Layer-1 validator; requires Phase 0 provider and privacy decisions. Blocks Phase 3 and Phase 4a.

    Phase 3: Safety and Review Workflow (3-4 weeks; overlaps Phase 2)¶

    Branch: feat/phase-3-safety-review Milestone: M3 - Parent approval gate enforced end to end Status: Planned

    Objective: Make the kids-facing guarantee real: no story reaches a child without a recorded guardian approval. See ADR-005.

    Deliverables (traced to Roadmap Phase 3):

    • Moderation pass: provider moderation API plus an independent LLM-reviewer pass, scored against per-age-band policy; any hit flags the nodes and forces human review. Safety hits always route to a person; never auto-publish.
    • Publish state machine with the guardian-only approval transition: draft -> generating -> auto_check -> in_review -> approved -> published -> archived, with needs_revision on auto-check failure.
    • Parent review surface: a guardian can read the full story, see flagged passages, and approve or send back. The review UI must make each approval achievable in a few minutes.
    • Provenance and audit on every published version: model, provider, prompt version, and approver ID persisted in storybook_version.
    • Authorization enforcement: child tokens cannot call approve or publish; IDOR negative tests green (child A cannot see child B's data; a guardian from another family cannot access a story).

    Acceptance criteria (from Roadmap Phase 3):

    • No story reaches a child profile without a recorded guardian approval (verified by attempting every transition path).
    • Adversarial briefs are flagged and cannot be auto-published.

    Quality gates:

    • 80% line / 70% branch coverage overall; 90% on publish state machine and authorization checks (critical paths)
    • All IDOR negative tests pass (child A/B cross-access, child calling approve/publish, cross-family guardian access)
    • Safety evaluation: adversarial concept briefs verified to flag moderation and route to human review; no auto-publish path
    • No high or critical security findings (Bandit, Semgrep, pip-audit)
    • Ruff clean; BasedPyright strict clean
    • Pre-commit green; all commits signed with conventional messages

    Dependencies: Requires Phase 2 generation and validation. Phase 3 may begin overlapping Phase 2 once the validation gate and orchestrator are stable.

    Phase 4a: Library and Profiles (part of 3-5 weeks; in the first release)¶

    Branch: feat/phase-4a-library-profiles Milestone: M4 - First usable release (generation + library) Status: Planned

    Objective: Make the first usable release shippable. A child sees only the stories permitted for their profile; a guardian can assign an approved generated story to one or more children. This is the first release line.

    Deliverables (traced to Roadmap Phase 4a):

    • Library browsing: GET /api/v1/library?profile_id={id} filters by published status, age band, and reading level cap of the requesting profile.
    • Per-child profiles: child_profile records with age_band, reading_level_cap, allowed_content_flags, and tts_enabled; enforced by the library API, never trusted from the client.
    • Minimal guardian path: guardian can view a generated story, approve it, publish it, and assign it to one or more child profiles, entirely through the UI.
    • Profile creation and management screens (guardian only).

    Acceptance criteria (from Roadmap Phase 4a):

    • A child sees only stories permitted for their profile (age band and reading level enforced).
    • A guardian can assign an approved generated story to one or more children.

    Quality gates:

    • 80% line / 70% branch coverage; 90% on library filtering and profile-enforcement paths
    • Integration tests: child cannot see unapproved or off-band story; guardian assigns story to profile and child library updates
    • No high or critical security findings
    • Ruff clean; BasedPyright strict clean
    • Pre-commit green; all commits signed with conventional messages

    Dependencies: Requires Phases 2 and 3. This phase closes the first release; Phase 4b and Phase 5 follow.

    Phase 4b: Editor, Engagement, and UX (post-first-release)¶

    Branch: feat/phase-4b-editor-ux Milestone: (post-release; no milestone number assigned) Status: Planned (post-release)

    Objective: Make authoring and reading pleasant beyond the minimum viable release. See ADR-002 (TTS via Web Speech API).

    Deliverables (traced to Roadmap Phase 4b):

    • Lightweight node editor: guardian can read a story as a playthrough and as a node list; edit a passage; re-roll a single branch (triggers repair pass); re-run validation. API: PATCH /api/v1/storybooks/{id}/versions/{v}/nodes/{node_id}.
    • Ending tracker: "3 of 7 endings found" display using the completion table; endings identified by stable ending.id so it survives prose edits.
    • Bookmarks: save-slot UI.
    • Read-aloud (TTS): Web Speech API (SpeechSynthesis), browser-native, targeted at the 8-11 band (Briella).

    Acceptance criteria (from Roadmap Phase 4b):

    • Concept to published through the UI alone including a small edit.
    • Read-aloud works for the youngest band.

    Quality gates:

    • 80% line / 70% branch coverage; 90% on node-edit and validation re-trigger paths
    • Playwright E2E: full edit-and-revalidate flow; TTS starts and reads passage
    • No high or critical security findings
    • Ruff clean; BasedPyright strict clean
    • Pre-commit green; all commits signed with conventional messages

    Dependencies: 4b can begin after the first release (Phases 0 through 4a merged). Does not block Phase 5.

    Phase 5: Hardening and Deploy (2-3 weeks; post-first-release)¶

    Branch: chore/phase-5-hardening Milestone: M5 - Hardened, deployed, restore-tested v1 Status: Planned (post-release)

    Objective: Production readiness on the homelab (or Azure Container Apps as the cloud alternative). See ADR-004.

    Deliverables (traced to Roadmap Phase 5):

    • Performance pass: node transition under 50 ms client-side from cache; library list API under 300 ms; first meaningful paint under 2 s on home wifi; validation under 2 s for a 200-node story; story blob under 500 KB for offline caching.
    • Offline-edge hardening: iOS PWA storage-eviction path tested end to end; navigator.storage.persist() called but not trusted on iOS; IndexedDB treated as cache only.
    • Accessibility: WCAG AA basics (contrast, focus order, scalable text).
    • Sentry wired on client and server.
    • Backups and restore drill: nightly Postgres dump and MinIO snapshot; a restore from backup succeeds in a drill.
    • Operator runbook: deployment, upgrade, and rollback steps.
    • Authoring guide: short non-technical guide for generating and approving stories.

    Acceptance criteria (from Roadmap Phase 5):

    • Deployed behind Pangolin with Authentik login; a restore from backup succeeds in a drill.
    • Performance targets met on a real device on home wifi.

    Quality gates:

    • All performance targets measured and met on a real device
    • Restore drill completes successfully and is documented in the runbook
    • Sentry events visible for both client and server test errors
    • WCAG AA contrast and focus-order checks pass
    • 80% line / 70% branch coverage maintained; no regressions
    • No high or critical security findings
    • Pre-commit green; all commits signed with conventional messages

    Dependencies: Requires Phases 1-4 (both 4a and 4b).

    6. Git Branch Strategy¶

    Branch names follow {type}/phase-{N}{suffix}-{description}. The primary branch is main.

    Phase Branch Type rationale
    Phase 0 chore/phase-0-foundations Infrastructure and decision lock; no user-facing feature
    Phase 1 feat/phase-1-schema-reader New user-facing capability (reader and schema)
    Phase 2 feat/phase-2-generation-gate New user-facing capability (generation pipeline)
    Phase 3 feat/phase-3-safety-review New user-facing capability (approval workflow)
    Phase 4a feat/phase-4a-library-profiles New user-facing capability (library and profiles)
    Phase 4b feat/phase-4b-editor-ux New user-facing capability (editor, TTS, tracker)
    Phase 5 chore/phase-5-hardening Infrastructure hardening; no new features

    Branch workflow¶

    # Start a phase
git checkout -b feat/phase-1-schema-reader

    Each phase branch merges to main via a PR after all acceptance criteria and quality gates pass. No direct pushes to main.

    7. Quality Gates (Definition of Done)¶

    These gates apply to every phase. Phase-specific gates are listed in each phase section above.

    Gate Standard Enforcement
    Line coverage >= 80% pytest --cov-fail-under=80 in CI
    Branch coverage >= 70% pytest --cov-branch in CI
    Critical paths and patch coverage >= 90% Condition evaluator, player state machine, validator, state machine, auth
    Linting Ruff: zero errors uv run ruff check .
    Formatting Ruff format uv run ruff format --check .
    Type checking BasedPyright strict: zero errors uv run basedpyright src/
    Frontend linting ESLint + Prettier clean CI step
    Security: SAST No high or critical Bandit, Semgrep managed rules, CodeQL
    Security: dependencies No high or critical uv run pip-audit; OSV-Scanner
    Security: containers No high or critical Trivy
    Secret scanning Clean detect-secrets
    SBOM Generated CycloneDX; images signed with Cosign
    Pre-commit Green pre-commit run --all-files before every commit
    Commits Signed + conventional git commit -S; conventional commit message format

    A feature is complete when code is reviewed and approved, tests are written and passing at the thresholds above, documentation is updated, no linting or type errors remain, security scans show no high or critical findings, and the commit is signed and merged to main via a PR.

    Source: Roadmap: Definition of Done and project CLAUDE.md.
